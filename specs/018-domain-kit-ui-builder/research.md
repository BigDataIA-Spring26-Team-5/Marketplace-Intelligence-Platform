## Decision 1: EnrichmentRulesLoader as the single source of truth for Tier 2

**Decision**: Extend `EnrichmentRulesLoader` (`src/enrichment/rules_loader.py`) with three new properties and update all five Tier 2 files to call them. No new YAML top-level keys invented beyond `text_columns`.

**Rationale**: `EnrichmentRulesLoader` already exists, is imported by `llm_enrich.py`, and exposes `safety_field_names()`, `deterministic_fields`, `llm_fields`. All five Tier 2 files just need to call the loader instead of reading module-level food constants. Keeps parameterization consistent and testable.

**New properties added to EnrichmentRulesLoader**:
- `enrichment_column_names: list[str]` — `[f.name for f in self.all_fields]`. Replaces `ENRICHMENT_COLUMNS = ["primary_category", "dietary_tags", "is_organic", "allergens"]` in `llm_enrich.py:16`.
- `llm_categories_string: str` — comma-separated string of `classification_classes` from the first LLM field. Replaces `CATEGORIES` in `llm_tier.py:36-40`.
- `text_columns: list[str]` — reads from YAML top-level `text_columns` key; falls back to `["product_name", "ingredients", "category"]`. Replaces hardcoded list in `deterministic.py:42`.

**`safety_field_names()` already exists** — returns names of all deterministic fields. No change needed. Used to replace `_SAFETY_FIELDS` in `llm_enrich.py:19` and `SAFETY_COLUMNS` in `guardrails.py:97`.

**VALID_CATEGORIES in `guardrails.py:112-117`** — replaced by `get_valid_categories(domain: str) -> frozenset` function that loads from `EnrichmentRulesLoader(domain).llm_fields[0].classification_classes`. Existing frozenset kept as fallback for when no domain pack is present.

**Alternatives considered**: (a) new `domain_config.yaml` top-level file per domain — rejected, more files to maintain; (b) config dict passed through all function calls — rejected, too much signature churn; (c) `src/` module-level `domain` global — rejected, not thread-safe.

## Decision 2: Kit generation uses a new LLM call, not `analyze_schema_node` directly

**Decision**: New module `src/ui/kit_generator.py` with `generate_domain_kit(domain_name, description, csv_content)` function. Makes a single LLM call using `get_orchestrator_llm()` to generate all three YAML files as strings. Does NOT invoke the LangGraph pipeline.

**Rationale**: `analyze_schema_node` maps a source to a KNOWN domain schema — it assumes `config/schemas/<domain>_schema.json` exists. Kit generation creates the domain schema from scratch. These are inverse problems. Reusing `analyze_schema_node` directly would require a dummy schema and would produce mapping YAML, not enrichment rules YAML. A focused new prompt is cleaner.

**LLM prompt inputs**: domain name, description, CSV header + 5 sample rows (as markdown table), Block contract reference, nutrition enrichment_rules.yaml as a few-shot example, block_sequence.yaml template.

**Alternatives considered**: (a) Reuse `derive_unified_schema_from_source()` — produces schema JSON, not enrichment rules; (b) Chain multiple LLM calls (one per file) — more cost, more latency, worse coherence; (c) Template-only (no LLM) — poor UX for enrichment field generation.

## Decision 3: Block scaffold generated by LLM, not a static template

**Decision**: New module `src/ui/block_scaffolder.py` with `generate_block_scaffold(domain_name, extraction_description) -> str`. LLM call using `get_orchestrator_llm()` with the `Block` base class contract and a food domain `extract_allergens.py` as few-shot example. Output validated as syntactically valid Python before display.

**Rationale**: Regex patterns for arbitrary domains (ICD-10, HCPCS, SKU formats, IATA codes) are domain-specific enough that a static template cannot produce useful patterns. LLM generates appropriate patterns from the extraction description. Validated with `ast.parse()` before display — no code is executed server-side.

**Security boundary**: Scaffold is returned as a string for download only. The file is NOT written to `custom_blocks/` by the system. The user manually places it. This keeps arbitrary code execution off the server's critical path.

**Alternatives considered**: (a) Static Jinja2 template with user-filled regex fields — no LLM needed but requires user to write regex; (b) Upload pre-written .py file — no generation, defeats the product purpose; (c) Auto-place in custom_blocks/ — security risk, rejected per spec.

## Decision 4: Streamlit UI as third mode in existing `app.py`

**Decision**: Add `"Domain Kits"` to the existing `st.radio("Mode", ["Pipeline", "Observability"])` on `app.py:615`. New function `_render_domain_kits_page()` dispatches to four sub-panels via `st.tabs()`. UI logic lives in `src/ui/domain_kits.py` (new module), keeping `app.py` as the router.

**Rationale**: Existing pattern (`app.py` routes to `_render_observability_page()`) is clean and consistent. No new Streamlit app, no new process. Domain Kits panel fits naturally alongside Pipeline and Observability as a management surface.

**`src/ui/domain_kits.py` tabs**:
1. **Generate Kit** — wizard (domain name, description, CSV upload → generate → review → commit)
2. **Block Scaffold** — extraction description → Python scaffold → security gate → download
3. **Preview / Validate** — resolve block execution plan, check enrichment_rules constraints
4. **Manage Kits** — list all domains, delete user-created ones

**Alternatives considered**: (a) Separate Streamlit app — rejected per spec assumption; (b) Inline in `app.py` — file already complex, keep routing only; (c) New sidebar page without tabs — less organized for 4 distinct sub-flows.

## Decision 5: Built-in domain detection by git-tracked files

**Decision**: A domain in `domain_packs/` is "built-in" if its directory contains any file tracked by git (`git ls-files domain_packs/<domain>/` returns non-empty). All others are "user-created". Checked at panel load time via `subprocess` call.

**Rationale**: Simple, accurate, uses authoritative source. Does not require a separate registry or metadata file. Works correctly even if a user manually adds files to `domain_packs/` without going through the UI.

**Alternatives considered**: (a) Hardcode list `["nutrition", "safety", "pricing"]` — fragile if new built-ins are added; (b) Metadata file `domain_packs/<domain>/.meta.yaml` with `type: built-in` — requires maintaining metadata; (c) Separate JSON registry — more infrastructure.

## Decision 6: `text_columns` YAML key added to `enrichment_rules.yaml` with backward-compatible default

**Decision**: Add optional top-level `text_columns: list[str]` to `enrichment_rules.yaml` schema. `EnrichmentRulesLoader.text_columns` reads it if present, falls back to `["product_name", "ingredients", "category"]`. Nutrition's existing `enrichment_rules.yaml` is NOT updated (fallback covers it — those columns exist in the nutrition schema).

**Rationale**: Avoids breaking the nutrition domain while enabling new domains to specify different source text columns. The fallback ensures existing tests pass without modification.

---

## Decision 7: `guardrails.py` VALID_CATEGORIES — add domain-aware getter, keep frozenset as fallback

**Decision**: Add `get_valid_categories(domain: str) -> frozenset` to `guardrails.py`. Internal guardrail functions that validate enrichment output call this getter when domain is known. Existing `VALID_CATEGORIES` frozenset retained as module-level default fallback for callers that don't supply domain.

**Rationale**: Guardrail functions are called from `analyze_schema_node` which has `domain` in state. Passing domain to the relevant validation functions is a localized change. The frozenset stays for backward compatibility — any caller not updated yet gets the food default, which is conservative (may reject valid non-food categories, but never incorrectly accepts them).

## Resolved Unknowns

| Unknown | Resolution |
|---------|-----------|
| How does `analyze_schema_node` derive source schema? | Reads `state["source_schema"]` — pre-computed from CSV headers + types before the node runs. Kit generator replicates this from uploaded CSV. |
| Does `llm_tier.py` make one call or batched calls? | Batched — `_LLM_BATCH_SIZE = 50`. Kit generation is separate; not affected. |
| How are custom blocks discovered at runtime? | `block_registry.py` scans `domain_packs/<domain>/custom_blocks/*.py` at registry init. Placing a scaffold file there takes effect on next Streamlit session load. |
| What validates generated YAML before commit? | `yaml.safe_load()` for syntax + structural check against expected top-level keys (`domain`, `fields` for enrichment_rules; `sequence` for block_sequence; `examples` for prompt_examples). |
| Is `EnrichmentRulesLoader` safe to call with a domain that has no enrichment_rules.yaml? | Yes — constructor logs a warning and sets `self.all_fields = []`. Fallback constants cover it. |
