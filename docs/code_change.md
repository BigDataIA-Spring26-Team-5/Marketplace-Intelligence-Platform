# ETL Pipeline Revamp: Block-Based Dynamic Schema Transformation

## Overview

This revamp refactors the ETL pipeline to use a unified **Block** system where:
- **Blocks** = schema transformations (type casts, format changes, column operations) — dynamically generated by Agent 2
- **Stages** = fixed domain logic (dedup, enrichment) — always present in pipeline
- The pipeline adapts dynamically to any input dataset schema

---

## Architecture Changes

### Before (Old Architecture)
```
FunctionRegistry (separate from BlockRegistry)
  - Stored simple Python functions
  - Agent 2 generated functions, not Blocks
  - Pipeline ran functions separately from blocks

BlockRegistry
  - Contained pre-built blocks only
  - No stage concept
  - No dynamic block loading
```

### After (New Architecture)
```
BlockRegistry
  - Pre-built blocks (normalization, DQ scoring)
  - Dynamically generated blocks (loaded from disk)
  - Stage definitions (dedup_stage, enrich_stage)
  - Auto-discovery at startup

Agent 2
  - Generates full Block classes (not functions)
  - Saves to src/blocks/generated/{domain}/
  - Blocks auto-loaded on next run
```

## New Concepts

### Block Types

| Type | Purpose | Example |
|------|---------|---------|
| **Pre-built** | Hand-written transformations | `StripWhitespaceBlock`, `LowercaseBrandBlock` |
| **Generated** | LLM-created for schema gaps | `TYPE_CONVERSION_product_name_acme_data` |
| **Stage** | Composite of multiple blocks | `dedup_stage`, `enrich_stage` |

### Generated Block Naming Convention
```
{GapType}_{TargetColumn}_{DatasetName}
```
- Example: `TYPE_CONVERSION_product_name_acme_data`
- Gap Types: `TYPE_CONVERSION`, `COLUMN_RENAME`, `COLUMN_DROP`, `COLUMN_CREATE`, `FORMAT_TRANSFORM`

### Stage Definitions

```python
_STAGES = {
    "dedup_stage": ["fuzzy_deduplicate", "column_wise_merge", "golden_record_select"],
    "enrich_stage": ["extract_allergens", "llm_enrich"],
}
```

## Pipeline Flow

```
┌─────────────────────────────────────────────────────────────────────┐
│ Agent 1: Orchestrator                                              │
│   ├── load_source_node: Load CSV, profile schema                  │
│   ├── analyze_schema_node: LLM gap analysis                        │
│   └── check_registry_node: Find existing blocks or mark misses   │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│ Agent 2: Code Generator                                            │
│   ├── generate_code_node: LLM generates Block classes              │
│   ├── validate_code_node: Syntax validation                        │
│   └── register_blocks_node: Save to src/blocks/generated/          │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│ Agent 3: Sequence Planner                                          │
│   └── plan_sequence_node: Order blocks (with stages)               │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│ Pipeline Runner                                                     │
│   └── Runs expanded sequence:                                     │
│       dq_score_pre → [generated] → normalization → dedup → enrich → dq_score_post │
└─────────────────────────────────────────────────────────────────────┘
```

## Default Block Sequence

```python
[
    "dq_score_pre",
    "__generated__",           # Agent 2 dynamic blocks
    "strip_whitespace",
    "lowercase_brand",
    "remove_noise_words",
    "strip_punctuation",
    "extract_quantity_column", # or keep_quantity_in_name for pricing
    "dedup_stage",             # expands to fuzzy_dedup + merge + golden_select
    "enrich_stage",            # expands to extract_allergens + llm_enrich
    "dq_score_post",
]
```

## Benefits of This Revamp

1. **Unified Registry**: Single source of truth for all transformations
2. **Persistent Blocks**: Generated blocks saved to disk, auto-loaded on restart
3. **Dynamic Adaptation**: Any new dataset schema can trigger new block generation
4. **Stage Clarity**: dedup/enrichment are clearly defined as composite stages
5. **Compute Efficiency**: Runner directly imports Block classes (lower overhead than wrappers)

## Notes

- FAISS is optional for enrichment (S2 KNN stage) - gracefully falls back if not installed
- Quarantine logic handles rows with nulls in required fields after enrichment
- Agent 3 can reorder blocks but cannot change stage execution order
