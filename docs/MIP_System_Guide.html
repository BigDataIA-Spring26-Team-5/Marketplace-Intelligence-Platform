1. Quick Start

Follow these steps to reproduce a full pipeline run end-to-end.

# Install dependencies (Python ^3.11, Poetry)
poetry install

# Configure environment: fill DEEPSEEK_API_KEY, ANTHROPIC_API_KEY, GROQ_API_KEY,
# GCS_*, BRONZE_BUCKET, SILVER_BUCKET, GOOGLE_CLOUD_PROJECT, UC2_PUSHGATEWAY_URL, ...
cp .env.example .env

# Canned 3-pass demo — USDA → FDA → FDA replay (showcases YAML cache hits)
poetry run python demo.py

# Primary CLI (checkpoint-aware; local CSV or gs:// URIs)
poetry run python -m src.pipeline.cli --source data/usda_fooddata_sample.csv --domain nutrition
poetry run python -m src.pipeline.cli --source gs://mip-bronze-2024/usda/2026/04/20/*.jsonl --mode silver
poetry run python -m src.pipeline.gold_pipeline --source off --date 2026/04/21

# Streamlit HITL wizard + Observability chatbot
poetry run streamlit run app.py           # http://localhost:8501

# (One-time) build FAISS KNN corpus
poetry run python scripts/build_corpus.py

# Bronze (JSONL) vs Silver (Parquet) row-count audit — metadata-only, no full scan
poetry run python scripts/count_rows.py
poetry run python scripts/count_rows.py --sources off usda openfda
poetry run python scripts/count_rows.py --date 2026/04/21
poetry run python scripts/count_rows.py --all-dates

# Docker alternative — full stack (Kafka, Airflow, Postgres, Prometheus, Pushgateway,
# Grafana, ChromaDB, Redis, MLflow)
docker-compose -p mip up -d

data/ and output/ are gitignored — drop data/usda_fooddata_sample.csv and data/fda_recalls_sample.csv in place before running demo.py.

Domain-Specific Unified Schema

Each domain (nutrition, safety, pricing, …) has its own domain-specific unified schema that is defined up-front, before any pipeline runs. The schema is the contract between the domain's source family and every downstream block — it fixes the target column set, types, and DQ/enrichment columns the pipeline will populate. Authoring the schema is a prerequisite, not a by-product of running the pipeline.

The pipeline currently ships with one domain-specific unified schema active: config/unified_schema.json, the nutrition/safety food-product schema shared by USDA, OpenFoodFacts (OFF), and openFDA — the three sources whose columns are naturally mappable to a common 14-column target (product_name, brand_name, ingredients, allergens, primary_category, DQ columns, …).

Amazon ESCI is a different shape entirely — it's a query/product relevance dataset with query, product_title, esci_label, ranking columns, no ingredients, no allergens. It cannot be forced through the nutrition unified schema without dropping most of its signal. Before running the pipeline against ESCI, a new domain-specific unified schema must be authored:

  1. Define the domain-specific unified schema up-front — e.g., config/unified_schema_pricing.json, with ESCI-appropriate columns (query, title, ranking, relevance label, …). This is a deliberate, human-authored decision made before any run, not something Agent 1 should be left to guess mid-pipeline.
  2. The first pipeline invocation against that domain bootstraps the JSON via FIRST_RUN_SCHEMA_PROMPT and derive_unified_schema_from_source() only if the file is absent — identical to how unified_schema.json was created from USDA on day one — but the canonical path is to commit the schema file first.
  3. Select the schema at run time by --domain (nutrition / safety → the food schema; pricing → the ESCI schema).

Today, the pricing domain is wired into the registry but its domain-specific unified schema file has not yet been authored; running the pipeline against ESCI data requires this up-front schema-definition step first. This is an intentional design point: one schema per domain keeps mappings tight and avoids ballooning the target column set with source-specific fields that are null for every other source. As additional domains are onboarded (e.g., restaurant menu data, retailer catalogs), each gets its own unified_schema_<domain>.json defined up-front, plus its own FAISS corpus namespace.

Three-Agent LangGraph

load_source → analyze_schema → [critique_schema?] → check_registry → plan_sequence → run_pipeline → save_output

Two conditional edges:

  • route_after_analyze_schema — skips critique_schema when (a) Redis returned a cached YAML mapping (cache_yaml_hit), or (b) with_critic=False (default off).
  • check_registry → plan_sequence — unconditional; Agent 3 always runs.

State flows through PipelineState (src/agents/state.py), a TypedDict(total=False) — fields are set incrementally; never assume a key is present.

4. Agent 1 — Schema Analyzer

File: src/agents/orchestrator.py::analyze_schema_node

Maps source columns to the 14-column unified schema (config/unified_schema.json). Emits an ordered list of schema operations; on first run also derives the unified schema itself via FIRST_RUN_SCHEMA_PROMPT, otherwise uses SCHEMA_ANALYSIS_PROMPT which excludes enrichment and computed columns from the mappable set.

Scope. Agent 1 maps source columns onto the domain-specific unified schema for the active --domain. The currently-shipped schema (food products, shared by USDA, OFF, and openFDA) is the only one authored today. A new source family whose columns don't fit this target (e.g., Amazon ESCI query/relevance data) requires a separate config/unified_schema_<domain>.json defined up-front before the first run, selected at run time by --domain. See “Domain-Specific Unified Schema” in §3.

Operation primitives (written to src/blocks/generated/<domain>/DYNAMIC_MAPPING_<source>.yaml):

PrimitiveSemantics
RENAME1:1 source → unified column rename
CASTType coercion (string, integer, float, boolean, datetime)
FORMATRegex-based cleanup or normalization
DELETEDrop source column
ADDCreate unified column with constant / null
SPLIT1-source → N-target (e.g., json_array_extract_multi)
UNIFYN-source → 1-target (e.g., coalesce, concat_columns)
DERIVECompute from other columns (expression, conditional_map, contains_flag)

Default model: claude-sonnet-4-5 (override with ORCHESTRATOR_LLM env var).

How to Invoke

5. Agent 2 — Critic

File: src/agents/critic.py::critique_schema_node

Audits Agent 1's operations against deterministic rules and returns revised_operations + critique_notes (rule / column / correction triples). Uses a reasoning model (anthropic/claude-sonnet-4-6) because Agent 1's JSON-mode output benefits from a slower, more deliberate reviewer.

Off by default. Enable with:

python -m src.pipeline.cli --source ... --with-critic

Skip conditions (handled by route_after_analyze_schema):

ConditionReason
cache_yaml_hit == TrueRedis returned a complete cached mapping; Agent 2/3 output already persisted
with_critic == FalseUser opted out; Agent 1 operations used as-is
operations == []Nothing to critique; no-op

6. Agent 3 — Sequence Planner

File: src/agents/graph.py::plan_sequence_node

Reorders the block pool returned by BlockRegistry.get_default_sequence(domain, unified_schema, enable_enrichment). Cannot add or remove blocks — any block the LLM drops is re-appended before dq_score_post by the guard code:

missing = [b for b in expanded_pool if b not in sequence]
if missing:
    if "dq_score_post" in sequence:
        idx = sequence.index("dq_score_post")
        for b in missing:
            sequence.insert(idx, b)
    else:
        sequence.extend(missing)

Pool shape depends on pipeline_mode:

ModePoolUsed by
fulldq_score_pre → __generated__ → cleaning blocks → dedup_stage → enrich_stage → dq_score_postCLI default, demo.py
silverdq_score_pre → __generated__ → cleaning blocks (no dedup, no enrichment)--mode silver
goldInvoked outside the graph; see src/pipeline/gold_pipeline.pypython -m src.pipeline.gold_pipeline

Cache-writer responsibility. plan_sequence_node is the sole writer for the full yaml cache entry. After all three agents run, it serializes column_mapping, operations, revised_operations, block_sequence, sequence_reasoning, enrichment_columns_to_generate, gaps, missing_columns, unresolvable_gaps, validation_profile, and the full YAML text under the schema-fingerprint key with a 30-day TTL.

8. Three-Tier Enrichment Cascade

File: src/blocks/llm_enrich.py

Cascading strategies of increasing cost; each row is resolved at the cheapest tier that produces a confident answer.

TierFileMechanismFields
S1 Deterministicsrc/enrichment/deterministic.pyRegex + keyword extraction from product textallergens, dietary_tags, is_organic, partial primary_category
S2 KNN Corpussrc/enrichment/embedding.py + corpus.pyFAISS IndexFlatIP over L2-normalized sentence-transformer vectorsprimary_category (only)
S3 RAG-LLMsrc/enrichment/llm_tier.pyGroq Llama-3.3-70B with top-3 S2 neighbors injected as RAG contextprimary_category (only)

Corpus key thresholds (src/enrichment/corpus.py, all overridable via env var):

VOTE_SIMILARITY_THRESHOLD     = 0.45     # neighbor vote eligibility
CONFIDENCE_THRESHOLD_CATEGORY = 0.60     # S2 resolves if ≥ this, else → S3
K_NEIGHBORS                   = 5        # FAISS top-k
MIN_CORPUS_SIZE               = 10       # single-row knn_search bootstrap
MIN_ENRICHMENT_CORPUS         = 1000     # batch-S2 short-circuit (embedding.py)
CORPUS_TTL_DAYS               = 90       # evict vectors unseen > this many days
MAX_CORPUS_SIZE               = 500_000  # hard size cap (oldest last_seen evicted first)
CHROMA_QUERY_CHUNK_SIZE       = 500      # max embeddings per ChromaDB HTTP call

Feedback loop. Both S2 and S3 append resolved rows back into the corpus. Persistent across runs: FAISS .bin is gitignored; corpus_metadata.json / corpus_summary.json are committed. The primary production corpus runs in ChromaDB (localhost:8000, collection product_corpus).

Ratio-Aware Augmentation & Two-Tier Eviction (spec 014)

At the start of every S2 pass, embedding_enrich invokes:

  1. evict_corpus(collection) — TTL-first (delete vectors where last_seen < now - CORPUS_TTL_DAYS), then size cap (delete oldest last_seen until count ≤ MAX_CORPUS_SIZE). Eviction runs in 500-ID batches; any ChromaDB failure logs a WARNING and returns without raising.
  2. augment_from_df(df, collection, unresolved_count, force_ratio_threshold=0.25) — if corpus_size / unresolved_count < 0.25, encodes S1-resolved rows and upserts them in 500-row chunks with last_seen = datetime.utcnow().isoformat(). Upsert semantics: existing vector IDs are not re-encoded, only their last_seen is refreshed.
  3. Short-circuit gate — if index.count() < MIN_ENRICHMENT_CORPUS after augmentation, S2 is skipped entirely and the pipeline falls through to S3.

Chunked KNN search. knn_search_batch now iterates embeddings in slices of CHROMA_QUERY_CHUNK_SIZE (default 500) rather than issuing one monolithic index.query call. A per-chunk failure fills that chunk's positions with (None, 0.0, []) and continues — the full batch never aborts on a single chunk. Progress is logged every 10 chunks at INFO level.

last_seen is load-bearing metadata. Every upsert path (add_to_corpus, build_seed_corpus, augment_from_df) writes an ISO-8601 last_seen timestamp. Re-upsert of the same _make_vector_id refreshes that timestamp, so frequently-recurring products stay fresh while one-off batch products age out naturally under TTL eviction.

Run-log surface. Gold pipeline enrichment_stats includes corpus_augmented (count upserted by augment_from_df) and corpus_size_after (total vector count after augmentation) — both 0 when ChromaDB is unreachable.

Safety Boundary — extraction-only safety fields

Hard rule: S2 and S3 touch only primary_category. The safety fields allergens, dietary_tags, is_organic are S1-extraction-only. They are never inferred by similarity or LLM, because confidently tagging a barley-containing product "gluten-free" is worse than a null.

Post-run guard in LLMEnrichBlock:

if any(row["<safety_field>"] differs from post-S1 state after S3):
    logger.warning("Safety invariant broken — investigate upstream")
If the warning fires, fix the upstream block; do not silence the warning.

How to Invoke

Part of enrich_stage in the default sequence; runs automatically in full and gold modes. Standalone:

from src.blocks.llm_enrich import LLMEnrichBlock
df = LLMEnrichBlock().run(df, config={"domain": "nutrition"})

9. Pipeline Runner & Integration

File: src/pipeline/runner.py::PipelineRunner

The integration layer that applies column_mapping and executes the block sequence. Streams input via CsvStreamReader in DEFAULT_CHUNK_SIZE = 10_000 row chunks. run_pipeline_node always calls run_chunked(); the in-memory run() is for tests and small synthetics only.

End-to-end steps orchestrated by graph.invoke():

 1. load_source        — Auto-detect delimiter, adaptive 5K-row sample, profile columns
 2. analyze_schema     — Agent 1 → column_mapping + operations + gaps
 3. critique_schema    — (optional) Agent 2 → revised_operations + critique_notes
 4. check_registry     — YAML lookup via BlockRegistry; HITL decisions for unresolvable gaps
 5. plan_sequence      — Agent 3 → ordered block_sequence; writes full YAML cache entry
 6. run_pipeline       — Stream chunks; apply column_mapping; execute blocks; compute DQ;
                         emit UC2 events (run_started, run_completed, quarantine, dedup_cluster)
 7. save_output        — Write CSV (full mode) or Parquet to gs://mip-silver-2024 (silver mode);
                         persist run log; push Prometheus metrics

Sequence expansion rules:

Sequence elementExpands to
"__generated__"DynamicMappingBlock for this source (from YAML)
"dedup_stage"fuzzy_deduplicate → column_wise_merge → golden_record_select
"enrich_stage"extract_allergens → llm_enrich

Quarantine rule. After enrichment, rows with any null in required_columns (minus excluded_columns and structurally-absent columns) are diverted to quarantined_df. Each row emits a quarantine Kafka event with a 16-character SHA256 row hash.

Quarantine output persistence. save_output_node writes quarantined_df alongside the primary output:

  • Silver mode → Parquet to gs://mip-silver-2024/<source>_quarantine/<YYYY/MM/DD>/ via GCSSilverWriter.
  • Full mode → output/<source>_quarantined.csv.

The destination URI is returned in PipelineState["quarantine_output_uri"] (src/agents/state.py).

Gold Pipeline — Reliability Hardening (spec 014)

_read_silver_parquet now skips blobs whose filename starts with sample (avoids pulling the 1K-row dev sample into full runs) and normalizes Arrow dtype nulls at the read boundary:

string_cols = [c for c in df.columns if str(df[c].dtype) == "string"]
if string_cols:
    df[string_cols] = df[string_cols].astype(object)

This single cast eliminates the downstream TypeError: boolean value of NA is ambiguous that previously crashed full-partition runs after ~4 hours. It applies uniformly to Arrow string and null dtype columns (e.g., brand_name on the branded/foundation/openfda Silver files).

run_gold_pipeline now constructs a CacheClient by default when none is passed in — Redis if reachable, SQLite fallback otherwise — so S3 LLM calls, embeddings, and dedup cluster keys are deduplicated across Gold re-runs. Runs are never blocked by Redis outages.

In src/enrichment/llm_tier.py, a new _safe_text(v) helper replaces every str(row.get(...) or "") site. It tolerates pd.NA, None, and float NaN without raising. S3 cache-hash computation and prompt construction both route through it.

Composite dedup blocking key (src/blocks/fuzzy_deduplicate.py). Replaced the 3-character name-only prefix with name[:4] + "_" + brand[:2]. This splits the former "cho" mega-block (12,038 chocolate products, 72M pairwise comparisons) across brands, cutting the OOM-threshold block count from 87 to under 20 on the OFF 783K-row partition. Null brand_name produces an empty brand prefix rather than a null-key collision; the existing valid_name_mask null-product-name guard is unchanged.

10. Redis Cache Layer

File: src/cache/client.py::CacheClient

Redis wrapper with a SQLite fallback (WAL-mode, at output/cache.db) when Redis is unavailable — ensures LLM cost savings persist across runs even on a dev box with no Redis.

Key format: pipeline:{prefix}:{sha256} where the SHA256 hash is over the canonical input (schema fingerprint, prompt+model, text, etc.).

PrefixTTLContentsHit effect
yaml30 daysFull Agent 1/2/3 output (column_mapping, operations, revised_operations, block_sequence, YAML text)Skips Agent 2 entirely via route_after_analyze_schema
llm7 daysS3 enrichment LLM responses (per prompt+model+text)Skips litellm.completion call
emb30 daysSentence-transformers embeddingsSkips model forward pass
dedup14 daysFuzzy dedup cluster keysReuses prior cluster assignments

TTLs overridable via CACHE_TTL_YAML, CACHE_TTL_LLM, CACHE_TTL_EMB, CACHE_TTL_DEDUP env vars.

Coherence rule. plan_sequence_node is the only writer for the yaml cache. If you add a field that Agent 1/2/3 produces, extend the cacheable dict there — otherwise replayed runs silently drop it.

Gold pipeline cache wiring. run_gold_pipeline(source_name, date, domain, cache_client=None, …) instantiates a CacheClient when none is passed; the instance is threaded into config["cache_client"] and reused by llm_enrich (LLM prefix) and fuzzy_deduplicate (dedup prefix). Redis unavailability logs a single WARNING and the pipeline proceeds via the SQLite fallback.

How to Invoke

cache.get_stats().log_all()   # hit/miss counters per prefix

16. Deviations from Specification

The following table documents where the implementation diverges from the specs in specs/, the rationale for each, and the impact on correctness.

AreaSpec SaysImplementation DoesRationale
Agent 2 invocation Agent 2 always runs Off by default; --with-critic enables it; also skipped when Redis YAML cache hits Keeps hot-path cost/latency down on replay runs; cached mapping already encodes Agent 2 output
Code generation Early spec (001) describes LLM-generated Python with subprocess sandbox YAML-only DynamicMappingBlock; src/agents/sandbox.py removed; function_registry/registry.json is vestigial [] Constitutional constraint per README contributor notes — YAML is the supported surface
function_registry/ directory Delete unused directory Kept on disk with empty registry Older checkpoints reference the path; keeping it prevents FileNotFoundError on resume; not read by any current code path
UC3 / UC4 modules Specs imply working hybrid-search and recommendation subsystems src/uc3_search/*, src/uc4_recommendations/* all raise NotImplementedError Scaffolding only; not wired into graph, CLI, demo.py, or app.py
UC2 placeholders anomaly_detection.py, dashboard.py as live modules Raise NotImplementedError; live anomaly detection lives in anomaly_detector.py (note the filename difference) Module rename mid-spec; legacy import paths preserved as stubs
Safety-field enrichment Spec implies LLM allergen inference as stretch goal Hard-blocked; LLMEnrichBlock warns if any S3-resolved row has allergens/dietary_tags/is_organic differing from post-S1 state False positives on safety are worse than nulls (e.g., mistakenly tagging barley-containing product "gluten-free")
Spec numbering collisions Unique numbered specs 006-production-deployment + 006-uc1-uc2-integration; 013-silver-bronze-pipeline + 013-silver-gold-pipeline 010-uc1-uc2-integration supersedes 006-*; the Gold pipeline spec is authoritative for 013-* (its header reads "Spec 012 — Gold Layer")

LLM routing DeepSeek throughout Multi-provider: Claude (orchestrator + critic), DeepSeek (legacy codegen getter), Groq Llama-3.3-70B (S3 enrichment), Groq Llama-3.1-8B (observability) Sonnet provides better JSON-mode stability; reasoning model for critic; Groq for enrichment cost + throughput
Metrics push path Single MetricsExporter Both _MetricsCollector (legacy) and MetricsExporter push in save_output_node Parallel paths during 011-spec rollout; legacy will be retired once dashboards fully migrated
Chunk size Not specified DEFAULT_CHUNK_SIZE = 10_000 in csv_stream.py, overridable via --chunk-size Balances FAISS batch efficiency, pandas memory, and UC2 event volume
Gold pipeline dtype tolerance Silver Parquet writers produce clean object dtype Gold _read_silver_parquet casts Arrow StringDtype / null columns to object at the read boundary; _safe_text in llm_tier.py tolerates pd.NA / None / float NaN Silver ETL still emits Arrow-null text columns for branded/foundation/openfda; Gold absorbs the defect rather than blocking until the Silver fix ships
Dedup blocking key Simple name-only prefix Composite key: name[:4] + "_" + brand[:2] 3-char name-only key produced 12K-row "cho" mega-blocks (72M pairwise comparisons); composite key drops OOM-threshold blocks from 87 to <20 on the OFF 783K partition

Corpus bootstrap threshold MIN_CORPUS_SIZE = 10 Batch-S2 uses MIN_ENRICHMENT_CORPUS = 1000 as a hard short-circuit; MIN_CORPUS_SIZE = 10 retained only for the single-row knn_search path At 10 vectors, S2 burned CPU encoding 400K rows for a corpus far too sparse to vote meaningfully; 1000-vector floor is empirically the break-even point
Corpus growth strategy Append-only Two-tier eviction: TTL-first (CORPUS_TTL_DAYS=90), then size cap (MAX_CORPUS_SIZE=500_000) by oldest last_seen; ratio-aware augmentation (force_ratio_threshold=0.25) from S1-resolved rows Prevents stale-taxonomy drift and unbounded vector-store growth; keeps S2 recall high on sparse-corpus runs
